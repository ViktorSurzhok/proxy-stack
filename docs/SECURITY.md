# Security notes — proxy-stack

## Модель угроз

Проект рассчитан на **личный** VPS и ограниченный круг клиентов (ваш браузер и Telegram). Это не корпоративный zero-trust perimeter.

## Tinyproxy

- По умолчанию доступ к HTTP-прокси ограничен **списком IP** (`Allow` в `tinyproxy.conf` + правила UFW). Это основной и самый предсказуемый способ защиты для браузера.
- **BasicAuth** в Tinyproxy поддерживается, но в браузерах качество поддержки HTTP-прокси с логином/паролем неоднородно (особенно в старых версиях и при смешении с HTTPS). Поэтому BasicAuth здесь **опционален** и дополняет, а не заменяет, сетевое ограничение по IP.
- Режим **open_warning** (`CONFIRM_OPEN_PROXY=I_UNDERSTAND_OPEN_PROXY_RISK`) создаёт фактический **open HTTP proxy** для всего интернета. Используйте только осознанно; возможны злоупотребления (сканирование, релей трафика, блокировки у хостера).

## MTProto (Telegram)

- Секрет MTProxy по сути является **общим ключом доступа** к прокси: любой, кто знает `server`, `port` и `secret`, может подключиться. Храните `secrets.env` и сводку как пароли.
- Образ `telegrammessenger/proxy` давно не обновлялся на Docker Hub; следите за новостями Telegram и при необходимости меняйте реализацию.
- Порт **443** часто занят веб-сервером. Скрипт при конфликте переключает MTProto на **8443** (и корректно формирует ссылки в summary).

## UFW

- При **первом** включении UFW скрипт выставляет `default deny incoming`, `allow outgoing`, `limit 22/tcp` и включает файрвол. Если SSH у вас **не на 22/tcp**, добавьте правило вручную **до** включения UFW или правьте процесс установки.
- Правила, помеченные `proxy-stack svc=`, удаляются при повторной установке / `reconfigure` / `uninstall`. Остальные правила UFW не трогаются.

## Секреты

- Файл `/opt/proxy-stack/secrets.env` (или `STATE_DIR`) должен иметь права **600** и не попадать в git.
- Не пересылайте сводку и секреты открытым текстом в публичные каналы.

## Логи

- `install.log` может содержать IP и параметры окружения. Держите права **600**.
